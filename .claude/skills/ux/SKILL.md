# UX Skill

Use UX methods to improve the real product flow, not to generate ritual deliverables with no implementation value.

## In DevZeros

- `/office/:projectId` is the primary runtime operator surface
- `/project/:projectId` is a secondary detail surface
- approvals, workflows, transcript clarity, and language consistency are product-critical UX concerns
- operator trust matters more than decorative delight

## Core Methods

- task-flow analysis
- journey mapping for studio, office, project, and approval flows
- heuristic review of friction, ambiguity, and state clarity
- usability testing for critical operator or customer actions
- accessibility review against WCAG 2.2 expectations

## What To Evaluate

- can the user understand the current state quickly?
- are next actions obvious?
- are blockers and approvals clearly surfaced?
- does language remain consistent with customer-selected locale?
- do loading, empty, and error states preserve confidence?
- can keyboard and assistive-technology users complete the flow?

## UX Heuristics

- reduce ambiguity before adding delight
- keep repeated controls and help mechanisms consistent across related surfaces
- do not make users re-enter information unnecessarily
- keep support and recovery paths visible when workflows fail or pause
- avoid interaction patterns that look clever but slow operators down

## Output Style

When giving UX guidance, keep it concrete:

- affected flow
- friction observed
- impact on task completion or trust
- recommended change
- how to verify the improvement
